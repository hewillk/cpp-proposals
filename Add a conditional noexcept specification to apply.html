Add a conditional noexcept specification to std::apply

    • Impact on the Standard
    • Implementation experience

Abstract

This paper proposes to add a noexcept-specification to std::apply.

Discussion

With the introduction of the C++23 zip family, apply began to rock and roll and appeared in a lot of its implementations.

For example, in [range.zip.transform.iterator], zip_transform_view::iterator::operator*()'s equivalent Effect is defined as: 

  
    return apply([&](const auto&... iters) -> decltype(auto) {
      return invoke(*parent_->fun_, *iters...);
    }, inner_.current_);
which expands the tuple contains iterators of views through apply and forwards them to the callable.

However, strictly speaking, this operator*() should not be done through apply. The reason is that it still has a noexcept(see below) specifier which is equivalent to noexcept(invoke(*parent_­->fun_­, *std​::​get(inner_­.current_­)...)), where Is is the pack 0, 1, …, (sizeof...(Views)-1), and unfortunately, the standard is very conservative with the noexcept specifications in the library specification, this makes apply lacks the noexcept specifier and becomes a non-noexcept function, however, since the standard also defines another equivalent to semantics of Effects of apply, this part of the semantics constitutes the effective noexcept specification of operator*().

But if we look at apply more closely, according to its Effects in [tuple.apply]:

  
    template<class F, class Tuple, size_t... I>
    constexpr decltype(auto) apply-impl(F&& f, Tuple&& t, index_sequence<I...>) {
      return INVOKE(std::forward<F>(f), get<I>(std::forward<Tuple>(t))...) // exposition only
    }

We can find that it uses index_sequence to expand get to extract tuple elements, and forward them to INVOKE together with callable, and because get is a noexcept function (except for the subrange overload, but the standard does not specify whether apply can be applied to subrange) and invoke is condition noexcept, there is no reason not to make apply transparently become condition noexcept.

In my opinion, invoke(f, args...) should be completely equivalent to apply(f, forward_as_tuple(args...)), and adding noexcept to apply can easily achieve this and make it more consistent with invoke.

Impact on the Standard

Since it is a pure change for apply, there will be no impact. For the zip family, apply can be used indeed for its implementation. This also makes it free to add noexcept specifier to functions that main implemented by apply.

Implementation experience

This proposal has been implemented by libstdc++ and libc++, but the libstdc++'s implementation is based on the premise that std::get will not throw an exception.

Proposed change

    1. Edit 20.5.5 [tuple.apply] as indicated:

      template<class F, class Tuple>
  constexpr decltype(auto) apply(F&& f, Tuple&& t) noexcept(see below);
      -1- Effects: Given the exposition-only function: 
      namespace std {
  template<class F, class Tuple, size_t... I>
  constexpr decltype(auto) apply-impl(F&& f, Tuple&& t, index_sequence<I...>) {     
                                                                        // exposition only
    return INVOKE(std::forward<F>(f), get<I>(std::forward<Tuple>(t))...);
  }
}
      Equivalent to: 
         return apply-impl(std::forward<F>(f), std::forward<Tuple>(t),
                     make_index_sequence<tuple_size_v<remove_reference_t<Tuple>>>{});
      -2- Remarks: Let I be the pack 0, 1, …, (tuple_size_v<remove_reference_t<Tuple>>-1). The exception specification is equivalent to: noexcept(invoke(std::forward<F>(f), get<I>(std::forward<Tuple>(t))...)).
